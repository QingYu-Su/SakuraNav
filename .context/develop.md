# SakuraNav - AI 开发规范

> 本文件定义了 AI 在协助开发/修复功能时必须遵守的规范。

## 代码结构

1. **单文件行数上限 1000 行** — 超过时必须进行合理的拆分。拆分必须遵循代码逻辑、有条理、满足高内聚低耦合的原则，优先考虑组件化（提取独立子组件、抽取工具函数、分离关注点等）。**禁止**通过删除注释、合并多行代码、压缩代码格式等取巧手段来强行满足行数限制
2. **目录结构整洁** — 任何目录下，要么只存放代码文件，要么只存放子目录，不要在同一目录中混放文件和子目录；新建文件必须放到职责对应的目录中

## 开发流程

1. **先读文档再开发** — 在开发具体需求时，必须先阅读 `docs/guide/` 中对应模块的开发文档（如 `auth.md`、`database.md`、`hooks.md` 等），了解清楚上下文和既有架构约定，再进行编码，避免随意开发而不遵循此前的代码规范
2. **禁止手动修改 `storage/` 目录** — `storage/` 目录在首次构建运行后会自动创建，包含数据库、上传文件等用户数据。在开发代码过程中，不要手动修改其中的文件
3. **文档站点独立** — `docs/` 目录下的 VitePress 文档站点是独立子系统，与主项目代码无耦合关系。修改主项目代码不需要考虑 VitePress 的构建逻辑，反之亦然

## 项目约定

遵循项目已有的架构模式，不要绕过它们：

1. **类型安全** — 禁止使用 `any`，新增数据结构必须在 `src/lib/base/types.ts` 中定义类型
2. **API 参数校验** — 新增或修改 API 接口时，必须在 `src/lib/config/schemas.ts` 中配套 Zod 校验 Schema
3. **数据库变更** — 修改表结构时必须同步更新 `schema.ts` 和 `migrations.ts`，并更新对应 Repository
4. **Repository 模式** — 数据访问必须通过 Repository 层（`src/lib/services/`），禁止在 API 路由中直接编写 SQL
5. **服务端安全** — 涉及 API Key、密码等敏感配置，必须通过 `server-only` 包保护，确保仅在服务端可访问

## 代码质量

3. **代码风格** — 代码要优雅、具备良好的鲁棒性，关键逻辑必须有注释
4. **日志记录** — 重要功能处必须添加日志，方便后续排错

## UI 改动

5. **多场景适配** — 涉及 UI 层的改动，必须检查在以下场景下的适配情况：
   - 明亮模式
   - 暗黑模式
   - 自定义壁纸（含高亮/暗色壁纸）
   - 移动端响应式布局
6. **自定义弹窗** — 需要弹窗交互的场景（确认操作、表单编辑、信息展示等），禁止使用浏览器内置弹窗（`alert()`、`confirm()`、`prompt()`），必须使用项目自定义的美化弹窗组件
7. **自定义提示文字** — 按钮、图标等元素需要 hover 提示时，禁止使用浏览器原生 `title` 属性，必须实现样式良好的自定义 Tooltip 组件（注意使用 `createPortal` 渲染到 `document.body`，避免父级 `transform`/`overflow` 干扰定位）
8. **颜色对比度** — 设置特殊颜色的文字和区域时，必须注重区分度和对比度，确保在明暗模式下均清晰可读，避免颜色过浅导致内容难以辨认
9. **移动端弹窗居中** — 所有独立弹窗（Dialog/Modal/Popover）在移动端必须垂直居中显示（`items-center`），禁止使用 `items-end` 配合 `sm:items-center` 的方式让移动端弹窗贴底。弹窗内容高度需控制在视口范围内（如 `max-sm:max-h-[XXdvh]`），确保弹窗不超出屏幕边界，不会出现内容被截断或需要外部滚动条的情况

## 功能改动

9. **改动隔离** — 修改功能时不得影响无关功能，保持改动专注，避免连锁副作用
10. **安全性** — 编写代码时必须注重安全性，特别是：
    - 用户认证、密码输入等敏感界面，需防备 CSRF、XSS、暴力破解等攻击
    - 网络请求中的身份校验和特殊参数校验，需在服务端严格验证，不可仅依赖前端
    - 避免在客户端暴露敏感信息，API 接口需做权限控制和输入消毒
    - 用户输入必须经过校验和转义，防止注入攻击
11. **功能独立性** — 开发功能时要保证其独立性与健壮性，不受外部代码变更的影响而失效。例如：新增一个保存按钮后，后续其他改动（如新增数据类型、修改数据结构）不应导致该按钮保存数据不完整或操作无效。每个功能模块应对自身的数据完整性负责，明确声明所依赖的数据契约，避免隐式耦合
12. **联动完整性** — 在某个组件下新增/删除/修改功能时，必须排查所有会读取或消费该组件数据的其他模块，确保它们均能正确工作。例如：新增卡片类型后，导出功能、搜索功能、排序功能等涉及该数据的地方都必须同步适配，不得遗漏
13. **数据安全与精简** — 涉及重要数据时必须做到防伪装和防篡改，确保服务端数据的完整性和一致性。数据库字段设计要精简，不得出现字段缺失或无用数据遗留/冗余存储。每次变更数据结构后，需同步检查相关 API 的请求/响应字段是否对齐
14. **资源清理** — 避免数据和资源在服务器上的冗余。当用户永久删除某个资源，或新资源覆盖了旧资源时，如果旧资源和数据已无任何引用和恢复的可能，必须在服务端彻底清理（删除文件、清除数据库记录），不得遗留无用数据占用服务器空间
15. **文件读写安全** — 涉及文件读取/写入时必须进行限流和安全性校验：校验文件格式是否正确、限制上传/导入的数据量大小避免过大数据冲击服务器、验证文件内容结构符合预期，防止恶意或格式错误的数据进入系统

## 交付验证

7. **必须通过检查** — 代码修改完毕后，必须执行以下命令并确保**零 warning、零 error**：
   ```bash
   npm run lint
   npm run build
   ```
   如果检查不通过，必须修复直到两者完全没有任何 warning 和 error，再回复开发者已完成。

## 清理

8. **清理中间文件** — AI 在处理过程中产生的临时/中间文件（如日志输出、调试文件等），如果未被 `.gitignore` 忽略，必须在交付前删除，避免污染仓库。对于明确可以安全删除的文件（如会话残留的日志、临时文件等），应直接删除，无需向用户建议手动清理。
9. **清理历史遗留** — 由于需求变更、功能迭代等因素，代码中可能存在已失效的废弃代码（未使用的变量/函数/导入、已无用处的条件分支等）和过时的注释。经确认可安全删除后，应直接清理，避免项目代码臃肿。注意：必须确认无任何引用和副作用后方可删除，不可误删仍有作用的代码。
