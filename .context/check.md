# SakuraNav - AI 提交前检查清单

> 在执行 `git commit` 前，AI 必须按照本清单逐项检查所有修改的代码。

## ⚠️ 检查原则

**检查的依据是 `git diff` 的实际变更内容，而非 AI 对话上下文的残留记忆。**

执行 `@command://check` 时必须：
1. **先运行 `git diff`** 获取本次所有变更的完整内容（包括 `--stat` 概览和逐文件 diff）
2. **逐个审阅 diff 中的每处变更**，对照下方检查项逐一验证，不得因「之前对话中已确认过」而跳过任何 diff 内容
3. **不得依赖对话上下文的回顾性总结来替代 diff 审阅** — 上下文可能遗漏或误述实际代码变更，只有 `git diff` 才是真实依据
4. 对于 diff 中涉及的每个文件，必要时使用 `read_file` 阅读完整上下文以准确判断

## 检查项

### 结构与规范

- [ ] 是否遵循 `.context/develop.md` 中的全部规范
- [ ] **代码拆分审查** — 不仅检查单文件是否超过 1000 行（超过则必须拆分），还要从整体改动出发评估代码结构：是否存在职责混杂、耦合过重的模块？即使未超 1000 行，若逻辑上拆分为独立组件/函数更符合高内聚低耦合，也应主动提出拆分建议。不得通过删除注释、合并多行代码、压缩格式等取巧手段强行凑行数
- [ ] 新增文件是否放到了职责对应的目录？是否存在文件与子目录混放
- [ ] 是否绕过了项目已有的架构模式（Repository、Zod、types.ts 等）

### 代码质量

- [ ] 代码是否具有鲁棒性（边界处理、异常捕获、空值防护）
- [ ] 代码是否足够优雅（逻辑清晰、无冗余、命名合理）
- [ ] 关键逻辑是否有注释
- [ ] 重要功能是否添加了日志

### 功能与 UI

- [ ] 改动是否专注，未影响无关功能
- [ ] UI 改动是否在明亮/暗黑/壁纸/移动端下均适配
- [ ] 是否使用了浏览器内置弹窗（`alert`/`confirm`/`prompt`），如有则替换为自定义弹窗
- [ ] hover 提示是否使用了浏览器原生 `title`，如有则替换为自定义 Tooltip
- [ ] 特殊颜色文字/区域的对比度是否足够，是否在明暗模式下均清晰可读
- [ ] 是否引入了 `any` 类型

### 安全性

- [ ] 用户认证/密码输入相关代码是否防备了常见攻击（CSRF、XSS、暴力破解等）
- [ ] API 接口是否做了服务端权限控制和输入校验，而非仅依赖前端
- [ ] 敏感信息是否避免在客户端暴露
- [ ] 用户输入是否经过校验和转义，防止注入攻击

### 验证

- [ ] `npm run lint` 是否零错误
- [ ] `npm run build` 是否零错误

### 清理

- [ ] AI 处理过程中产生的临时/中间文件是否已清理（未被 `.gitignore` 忽略的无用文件）。对于明确可以安全删除的文件（如会话残留的日志、临时文件等），应直接删除，无需向用户建议手动清理
- [ ] 本次改动涉及的文件中，是否存在因需求变更而遗留的废弃代码（未使用的变量/函数/导入、无用条件分支等）或过时注释？确认无引用和副作用后应直接清理

## 评分

对本次修改整体打分（0-10 分）：

| 维度 | 分数 (0-10) |
|:-----|:------------|
| 规范遵循 | |
| 代码质量 | |
| 改动专注度 | |
| 鲁棒性 | |
| **总分** | **/ 40（换算为 10 分制）** |

> 换算公式：总分 ÷ 4 = 最终评分
>
> **最终评分 ≥ 8 分** → 执行以下操作：
> 1. 根据本次代码变更，同步更新 `docs/` 下的开发文档（DEVELOPMENT.md、DOCKER.md 等），**不包含 README.md 、README_EN.md 和 CHANGELOG.md**。注意：只有当本次变更涉及功能、架构、代码文件的**新增、变动或删除**时才需要更新文档；如果只是修复代码与已有文档不一致的 bug，则无需更新文档。更新文档时，若发现已有内容描述的功能/特性已被替换或删除（如废弃的配置项、已移除的 API、旧版目录结构等），应直接删除，保持文档与项目实际状态一致，避免文档臃肿
> 2. 建议提交 commit，并给出建议的 commit 信息。commit 信息须遵循 [Conventional Commits](https://www.conventionalcommits.org/) 规范，格式为 `<type>(<scope>): <subject>`，type 包括 `feat`、`fix`、`refactor`、`docs`、`style`、`test`、`chore` 等，scope 为可选的模块/功能范围，subject 为简短的中文描述（不加句号）
>
> **最终评分 < 8 分** → 列出需要修复的问题，修复后重新检查
